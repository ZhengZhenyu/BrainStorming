## 1. 背景与研究动机

### 1.1 开源度量的核心挑战

开源社区的健康度评估本质上是一个多维度、动态变化的复杂问题。与商业产品的评估不同，开源项目同时具备**技术产品**和**社会组织**的双重属性，这导致单一维度的度量往往失真。

业界在开源度量领域面临的主要挑战包括：

**度量维度的取舍困境**：技术质量、社区活跃度、商业影响力、安全合规往往相互独立甚至相互冲突。一个技术质量极高但社区冷清的项目，和一个社区活跃但代码质量参差不齐的项目，哪个更"健康"？不同利益相关方的答案截然不同。

**阶段性与可比性的矛盾**：处于孵化期的项目与成熟项目用同一套指标衡量，本质上是在比较"婴儿与成人的体型"，结论必然误导。然而管理多个项目的运营团队又需要某种可横向比较的基准。

**量化与定性的边界**：贡献者的归属感、社区的包容性文化、技术决策的透明度，这些对社区长期健康至关重要的因素，至今缺乏有效的量化手段。

**数据可及性**：许多关键指标（如贡献者的真实组织归属、代码的实际生产使用情况）难以自动化获取，依赖手动调研或平台授权，限制了大规模度量的可行性。

### 1.2 本报告的研究范围

本报告聚焦以下七个业界主流框架，从理念、指标体系、工具支持、适用场景和局限性五个维度进行详细解析：

| 框架 | 主导方 | 主要关注领域 |
|------|--------|-------------|
| CHAOSS | Linux 基金会 | 开源社区全面健康度 |
| OSS-Compass | 华为 / 开放原子等 | 中文开源生态健康度 |
| Apache 成熟度模型 | Apache 软件基金会 | 社区治理与毕业评估 |
| CNCF 分级标准 | CNCF / Linux 基金会 | 云原生项目生命周期 |
| ISO/IEC 25010 | ISO / IEC | 软件产品质量 |
| DORA Metrics | Google DevOps Research | 软件交付效能 |
| OpenSSF Scorecard | OpenSSF / Google | 开源安全实践 |

## 2. 框架概览与选型建议

### 2.1 框架定位地图

各框架并非竞争关系，而是从不同切面描述同一项目的不同侧面。以下定位地图有助于理解各框架的适用场景：

```
                    【社区/人】
                        ↑
              CHAOSS    |    OSS-Compass
              Apache    |    CNCF
                        |
【过程/工程】←————————————+————————————→【产品/技术】
                        |
              DORA      |    ISO/IEC 25010
              OpenSSF   |
                        ↓
                    【安全/合规】
```

### 2.2 快速选型建议

| 使用场景 | 推荐框架 | 理由 |
|---------|---------|------|
| 评估国内开源项目整体健康度 | OSS-Compass + CHAOSS | 数据源更完整，与国内平台（Gitee等）集成好 |
| 评估项目是否具备捐献基金会条件 | Apache 成熟度模型 / CNCF | 直接对应基金会评审标准 |
| 评估工程团队交付效能 | DORA | 权威度高，行业 benchmark 数据丰富 |
| 评估软件产品质量（合规/采购场景）| ISO/IEC 25010 | 国际标准，法律合规认可度高 |
| 评估开源依赖的供应链风险 | OpenSSF Scorecard | 自动化程度高，CI/CD 可直接集成 |
| 跨基金会、跨语言的多项目横向对比 | CHAOSS | 中立性最强，社区最广 |

## 3. CHAOSS — 业界最权威的社区健康度框架

### 3.1 背景与发展历程

CHAOSS（Community Health Analytics Open Source Software）由 Linux 基金会于 **2017 年正式成立**，起源于 2016 年 CHAOSScon 会议上学术界和工业界对"如何客观评估开源社区健康状况"的共同讨论。

核心参与方包括 Mozilla、Red Hat、GitHub、Google、University of Nebraska 等，目前已发布超过 **80 个标准化指标定义**，是业界引用最广泛的开源健康度参考体系。

CHAOSS 的核心设计哲学是：**只定义指标含义，不强制工具和实现方式**。这使其成为"语言"而非"系统"，具备极强的通用性。

### 3.2 五大工作组与核心指标

#### 工作组一：Evolution（演进）

关注社区的活跃程度和成长轨迹，是最基础的健康信号。

**核心指标详解：**

| 指标名称 | 定义 | 健康信号 | 预警信号 |
|---------|------|---------|---------|
| Code Changes（代码变更量） | 单位时间内合入的 commit 数量 | 持续稳定增长 | 连续下降 > 3 个月 |
| Code Change Lines（代码变更行数） | 合入代码的行数（增删合计）| 与 commit 数正相关 | 变更行数骤降但 commit 数正常（可能是无效提交）|
| Issue Resolution Duration（Issue 解决时长）| 从 Issue 创建到关闭的平均时长 | < 30 天（活跃项目）| > 90 天（响应迟钝）|
| New Contributors（新贡献者数）| 首次提交被合入的新贡献者数量 | 每季度稳定增长 | 新贡献者长期为零 |
| Change Request Closure Ratio（PR 关闭率）| 已关闭 PR / 总 PR 数 | > 80% | < 60%（积压严重）|

**实践注意**：Code Change Lines 单独使用易被刷数据，应与 Code Changes 数量结合使用，并排除自动化机器人的提交。

---

#### 工作组二：Diversity & Inclusion（多样性与包容性，D&I）

D&I 是 CHAOSS 中最具人文关怀、也是最难量化的工作组，于 2018 年独立成组，反映了开源社区对可持续发展的深层次认知：**一个只能吸引单一背景贡献者的社区，长期来看是脆弱的**。

**核心指标详解：**

| 指标名称 | 定义 | 衡量维度 |
|---------|------|---------|
| Contributor Location Diversity | 贡献者的地理分布熵值 | 时区多样性、地区均衡性 |
| Organizational Diversity | 贡献组织数量及 HHI 指数 | 防止单一公司主导 |
| New Contributor Onboarding（新贡献者引导）| 从首次 PR 到第二次贡献的转化率 | 贡献门槛与引导质量 |
| Documentation Accessibility | 文档的语言数量、可读性评分 | 非英语贡献者的参与门槛 |
| Response Time to Newcomer Questions | 对新贡献者问题的首次响应时长 | 社区友好程度 |

**深度洞察**：CHAOSS D&I 工作组提出了 **"包容性设计"** 概念——度量本身不应成为歧视工具。例如，用性别作为多样性指标时需要注意数据收集的伦理边界，GitHub 个人资料中的性别信息仅供用户自愿填写。

---

#### 工作组三：Risk（风险）

风险工作组关注**项目的可持续性威胁**，核心问题是：如果某些关键人或关键组织退出，项目是否仍能运转？

**核心指标详解：**

**Bus Factor（巴士系数）**
- 定义：如果 N 个关键贡献者同时离开，项目将陷入无法继续的危机，此 N 即为巴士系数
- 计算方式：找出覆盖 50% 代码贡献的最少贡献者数量
- 健康基准：≥ 3（避免"被巴士撞到"导致项目失败的风险）
- 与你的"大象系数"区别：大象系数关注组织层面，巴士系数关注个人层面，两者互补

**Elephant Factor（大象系数）**
- 定义：覆盖 50% 代码贡献所需的最少贡献**组织**数量
- 健康基准：≥ 2（Apache）；≥ 3（CNCF 毕业要求）
- 实践注意：需要识别并合并同一组织的多个账号（如华为员工使用个人账号提交）

**License Coverage（License 覆盖率）**
- 定义：代码库中有明确 License 声明的文件比例
- 重要性：License 不清晰是商业采用的最大法律障碍
- 健康基准：> 95%

**Libyears（依赖老化程度）**
- 定义：项目所有依赖与其最新版本的时间差之和（以年为单位）
- 健康基准：总 Libyears < 12（即平均每个依赖不超过 1 年未更新）

---

#### 工作组四：Value（价值）

Value 工作组是最新且最具争议的工作组，尝试回答"开源项目产生了多少经济和社会价值"这一难题。

**核心指标详解：**

| 指标 | 定义 | 评估难点 |
|------|------|---------|
| Downstream Dependencies（下游依赖数）| 依赖本项目的其他开源项目数量 | 间接依赖难以完整统计 |
| Academic Citations（学术引用）| 学术论文中引用项目的次数 | 仅适用于研究导向型项目 |
| Job Postings（招聘关联度）| 招聘信息中提及项目技术的数量 | 反映商业生态活跃度 |
| Social Media Mentions（社媒提及量）| 社交媒体的讨论量和情感分析 | 噪音较大，需过滤 |

**实践评价**：Value 工作组指标在实际运营中落地难度最大，多数团队将其作为参考维度而非核心 KPI。

---

#### 工作组五：Common（通用基础）

定义所有其他工作组的公共基础概念，确保跨工作组指标的一致性。

**最重要的基础定义：**

**贡献者（Contributor）的范畴定义**：CHAOSS 明确指出，贡献者不仅限于代码提交者，还应包括：
- Issue 提交者和解决者
- PR 审查者（Review）
- 文档贡献者
- 社区活动组织者
- 翻译贡献者

这一宽泛定义对运营实践的意义在于：**代码提交量不能代表社区的全部活跃度**，运营型、文档型贡献同样需要被度量和激励。

### 3.3 CHAOSS 工具生态

| 工具 | 定位 | 特点 |
|------|------|------|
| **GrimoireLab** | CHAOSS 官方数据采集与分析平台 | 支持 Git、GitHub、Jira、Slack 等 30+ 数据源，开源 |
| **Augur** | 美国内布拉斯加大学研究项目 | 学术研究向，数据深度强 |
| **Bitergia Analytics** | 商业化 CHAOSS 实现 | 企业级支持，有托管服务 |
| **LFX Insights** | Linux 基金会旗下项目 | 专为 LF 孵化项目设计 |

### 3.4 局限性与批评

- **指标数量过多**：80+ 指标对于大多数团队来说无法全量落地，缺乏优先级指引
- **缺乏行业 Benchmark**：CHAOSS 定义了"什么是指标"，但较少给出"多少才算好"的行业基准
- **实施成本高**：GrimoireLab 的部署和维护需要较强的技术能力
- **定性指标缺失**：对社区文化、决策透明度等软性因素覆盖不足

## 4. OSS-Compass（开源指南针）— 最贴近中文生态的度量平台

### 4.1 背景与定位

OSS-Compass 于 **2022 年底正式发布**，由华为、开放原子开源基金会、北京大学、中国科学院软件研究所等联合发起，是目前国内最系统的开源社区评估平台。

其核心价值主张是：**在 CHAOSS 框架基础上，结合中国开源生态特点（Gitee、Gitea 等平台），提供本土化的度量服务**。

### 4.2 三维度评估模型

OSS-Compass 将所有指标归纳为三个一级维度：

#### 维度一：协作开发指数（Collaboration Development Index）

衡量项目的代码协作质量，核心问题是"贡献流程是否健康运转"。

**二级指标：**

| 指标 | 定义 | 权重说明 |
|------|------|---------|
| 代码提交频率 | 过去 90 天平均每周 commit 数 | 基础活跃度信号 |
| 贡献者数量 | 过去 90 天有效贡献者数量 | 衡量参与广度 |
| 提交评论频率 | Code Review 评论的频率 | 衡量协作质量，非数量 |
| 代码合入比率 | PR 合入数 / PR 总数 | 反映维护团队的响应能力 |
| Issue 完成比率 | 已关闭 Issue / 总 Issue 数 | 需求响应效率 |
| Issue 响应时间 | Issue 首次获得回复的平均时长 | 社区活跃程度的直接体现 |

#### 维度二：社区服务与支撑（Community Service & Support）

衡量社区对用户和贡献者的服务质量，核心问题是"社区是否友好、易于参与"。

**二级指标：**

| 指标 | 定义 | 实践意义 |
|------|------|---------|
| 更新版本数量 | 过去 12 个月的正式发版次数 | 稳定的发版节奏建立用户信任 |
| 文档完善度 | README、贡献指南、行为准则等关键文档的完备性 | 降低贡献者的引导成本 |
| 维护者响应率 | 维护者对 PR / Issue 的响应比例 | 核心团队的投入程度 |
| Bug 修复时长 | 从 Bug 报告到修复合入的平均时长 | 工程质量保障能力 |

#### 维度三：社区活跃度（Community Activity）

衡量社区的整体生命力，核心问题是"社区是否在持续生长"。

**二级指标：**

| 指标 | 定义 | 计算周期 |
|------|------|---------|
| 活跃贡献者数 | 过去 90 天有提交行为的贡献者数 | 滚动 90 天 |
| 贡献组织数 | 有贡献行为的不同组织数量 | 滚动 90 天 |
| 代码变更行数 | 增加行数与删除行数之和 | 滚动 90 天 |
| 关注者增长率 | Star / Watch 的增长速率 | 月度同比 |

### 4.3 综合健康评分机制

OSS-Compass 将三个维度通过加权平均生成一个 **0-100 的综合健康分**，权重可配置，默认权重为：

- 协作开发指数：40%
- 社区服务与支撑：30%
- 社区活跃度：30%

**重要说明**：OSS-Compass 官方不建议仅凭综合分进行项目排名，推荐结合三维度雷达图做结构性分析——一个总分 70 分的项目，可能是三个维度都是 70（均衡型），也可能是两个 90 一个 30（偏科型），运营策略截然不同。

### 4.4 与 CHAOSS 的差异对比

| 对比维度 | CHAOSS | OSS-Compass |
|---------|--------|-------------|
| 数据源覆盖 | GitHub 为主 | GitHub + Gitee + Gitoa |
| 中文社区支持 | 较弱 | 强（原生支持） |
| 实施门槛 | 高（需自建 GrimoireLab）| 低（SaaS 平台，开箱即用）|
| 指标标准化程度 | 极高（行业公认）| 中等（仍在演进中）|
| 学术引用度 | 高 | 中（较新）|
| 定制化能力 | 高 | 中 |

### 4.5 局限性

- 平台较新，指标体系仍在快速迭代，稳定性不及 CHAOSS
- 综合评分的权重设计缺乏公开的学术论证
- 对非 Git 托管项目（如部分嵌入式开源项目）支持有限

## 5. Apache 成熟度模型 — 社区治理的黄金标准

### 5.1 背景与核心理念

Apache 软件基金会（ASF）成立于 1999 年，管理超过 350 个顶级项目，是开源治理领域经验最丰富的基金会。

Apache 成熟度模型（Apache Project Maturity Model）并非专门的度量工具，而是 **ASF 用于评估孵化项目是否具备"毕业"资格的综合评审框架**。其价值在于：经过 20+ 年实践验证，代表了开源社区治理的最佳实践集合。

核心理念：**"Community Over Code"**——一个拥有健康社区的平庸项目，长期价值远超一个技术卓越但社区孱弱的项目。

### 5.2 六大评审维度

#### 维度 CD：代码（Code）

| 评审项 | 标准 |
|-------|------|
| CD10 | 项目产出软件，而非服务（SaaS 不适用 Apache 模型）|
| CD20 | 代码在 ASF 管理的代码仓库托管 |
| CD30 | 所有源代码均有 ASF 批准的开源 License |
| CD40 | 项目可以不依赖任何 Apache 外部服务运行 |
| CD50 | 项目的版本发布使用 ASF 代码签名 |

#### 维度 LI：License 与版权（Licenses & Copyright）

| 评审项 | 标准 |
|-------|------|
| LI10 | 代码库的 License 状态清晰，无歧义 |
| LI20 | 所有代码的版权归属明确（IP 清晰）|
| LI30 | 三方依赖的 License 兼容 Apache License 2.0 |

这是 Apache 模型中**法律风险最高**的维度，也是很多孵化项目最容易出问题的地方。尤其是历史代码的 IP 溯源，常常需要大量人工清理。

#### 维度 RE：发版（Releases）

| 评审项 | 标准 |
|-------|------|
| RE10 | 发版由社区批准（非个人决定），通过邮件列表投票 |
| RE20 | 发版产物是源码包（二进制包可选），而非仅二进制 |
| RE30 | 发版产物有 PGP 签名和 SHA 校验 |
| RE40 | 新版本发布遵循项目声明的版本策略（如语义化版本）|
| RE50 | 有明确的发版节奏（周期性或按里程碑）|

#### 维度 QU：质量（Quality）

| 评审项 | 标准 |
|-------|------|
| QU10 | 项目有公开可访问的 Bug 追踪系统 |
| QU20 | 项目有持续集成（CI）系统 |
| QU30 | 项目有安全漏洞报告渠道（非公开）|
| QU40 | 项目对已知安全漏洞有响应流程 |

#### 维度 CO：社区（Community）

| 评审项 | 标准 |
|-------|------|
| CO10 | 项目有 3 名以上活跃的 Apache Committer |
| CO20 | 贡献者来自至少 **3 个不同的组织**（核心判断项）|
| CO30 | 有新的 Committer 被授权（显示权力下放）|
| CO40 | 没有单一组织控制超过 **50%** 的代码决策权 |
| CO50 | 项目决策在公开邮件列表上进行（透明治理）|
| CO60 | 项目有明确的 CONTRIBUTING 文档 |
| CO70 | 有公开的路线图或开发规划 |

**CO 维度是整个模型的核心**，它体现了 Apache 的根本信念：代码可以被分叉，但健康的社区无法被复制。一个真正属于"社区"而非某家公司的项目，才具有长期可持续性。

#### 维度 CS：共识构建（Consensus Building）

| 评审项 | 标准 |
|-------|------|
| CS10 | 项目的技术决策通过社区讨论达成（非独裁）|
| CS20 | 项目使用"懒共识"（Lazy Consensus）等成熟决策机制 |
| CS30 | PMC（项目管理委员会）成员多元化 |
| CS40 | 项目无单点决策依赖（无"仁慈独裁者"风险）|

### 5.3 对运营实践的启示

Apache 模型对于**尚未捐献基金会但希望建立健康治理结构**的项目同样有参考价值。以下是可直接借鉴的实践：

1. **邮件列表决策透明化**：重要技术决策应留有公开可查的讨论记录，而非微信群/钉钉群的私下决定
2. **发版投票机制**：引入社区投票（哪怕是非正式的 +1/-1 机制），提升版本决策的社区归属感
3. **三组织原则**：将"贡献者来自至少 3 个不同组织"作为从成长期迈向成熟期的门槛指标

### 5.4 局限性

- 主要面向 ASF 孵化流程设计，对非 Apache 项目的普适性有限
- 邮件列表驱动的治理模式在快速迭代的现代开源社区中显得略为滞后
- 对中国开源文化（更多使用即时通讯工具决策）的适配性不足

## 6. CNCF 项目成熟度分级 — 云原生生态的阶段标准

### 6.1 背景与定位

Cloud Native Computing Foundation（CNCF）成立于 2015 年，隶属于 Linux 基金会，是 Kubernetes、Prometheus、Envoy 等云原生标志性项目的家园。

CNCF 项目成熟度模型（Project Maturity Levels）是目前**与社区生命周期结合最紧密**的基金会分级标准，也是本报告中与"分阶段度量"理念最契合的参考框架。

### 6.2 三级成熟度体系

#### 第一级：Sandbox（沙箱）

**定位**：早期项目，技术潜力已经显现，但社区尚未形成

**准入标准：**
- 项目处于云原生领域，与 CNCF 技术愿景对齐
- 代码已开源，有基本的 README 和构建说明
- 有至少 2 名维护者（Maintainers）
- 通过 CNCF TOC（技术监督委员会）的初步技术审核

**度量侧重：**
- 技术可行性（能否解决真实问题）
- 代码库基础质量（CI/CD 是否存在）
- 维护者活跃度

**类比你的框架**：对应**孵化期**，此阶段重点不是社区规模，而是技术方向是否正确、基础工程规范是否建立。

#### 第二级：Incubating（孵化中）

**定位**：项目已证明商业价值，正在建立多元化社区

**准入标准（核心）：**
- 至少 **3 个独立的最终用户（End Users）** 在生产环境使用（关键）
- 贡献者来自 **至少 2 个不同组织**
- 有清晰的版本发布流程
- 安全漏洞响应机制已建立
- 通过 CNCF 安全审计（对于核心基础设施类项目）

**度量侧重：**
- 生产采用证明（POC 不算数，需要是生产环境）
- 贡献者多元化进展
- 工程质量基准（CI、测试覆盖率、安全审计）

**类比你的框架**：对应**成长期**，CNCF 特别强调"生产使用证明"的重要性——有多少人用比有多少 Star 更重要。

#### 第三级：Graduated（毕业）

**定位**：项目已形成成熟的自治社区，具备长期可持续性

**准入标准（核心）：**
- 生产用户数量显著增加（通常 > 5 个大型最终用户）
- 贡献者来自 **至少 3 个不同组织**（多数毕业项目远超此标准）
- **Elephant Factor ≥ 2**（最大单一贡献组织的代码比例 < 50%）
- 有运作良好的治理结构（Governance 文档公开）
- 通过第三方安全审计
- 采用了 DCO（Developer Certificate of Origin）或 CLA

**度量侧重：**
- 社区自治能力（是否脱离创始公司主导）
- 长期维护承诺
- 生态系统成熟度

**类比你的框架**：对应**成熟期**，CNCF 毕业标准中"Elephant Factor ≥ 2"这一要求与你的大象系数指标直接对应。

### 6.3 CNCF 特色：技术雷达与年度调查

除成熟度分级外，CNCF 还发布：
- **CNCF 技术雷达**：基于社区调查，展示云原生技术的采用成熟度（评估/试用/采用/放弃）
- **CNCF 年度调查**：调查云原生技术的生产使用率，为项目价值评估提供数据基础

### 6.4 对运营实践的关键启示

**"3 个生产用户"原则**：CNCF 要求生产用户而非仅仅测试用户，这对于评估项目的真实影响力极为重要。建议在成长期指标中加入"生产环境采用案例数"作为补充。

**治理文档公开化**：毕业项目必须有公开的 GOVERNANCE.md，明确 PMC 构成、决策流程、冲突解决机制。这是社区治理成熟度的重要外部信号。

### 6.5 局限性

- 仅适用于云原生领域，对其他技术领域（如 AI、嵌入式、前端）参考价值有限
- "3 个生产用户"的验证依赖用户自愿公开，存在统计偏差
- 申请和审批流程耗时，不适合快速评估

## 7. ISO/IEC 25010（SQuaRE）— 软件产品质量国际标准

### 7.1 背景与定位

ISO/IEC 25010 是**软件质量领域最权威的国际标准**，属于 SQuaRE（System and Software Quality Requirements and Evaluation）系列标准的核心部分，于 2011 年正式发布（最新版本 25010:2023 于 2023 年更新）。

与其他框架不同，ISO/IEC 25010 是**法律和采购层面认可的正式国际标准**，在以下场景中具有不可替代性：
- 政府或大型企业采购软件时的评估依据
- 软件产品合规认证
- 开源软件进入关键基础设施的准入评估

### 7.2 产品质量模型：8 大特性

ISO/IEC 25010 定义了两个质量模型，其中**产品质量模型**对开源软件评估最为相关：

#### 特性一：功能适合性（Functional Suitability）

| 子特性 | 定义 |
|--------|------|
| 功能完备性 | 软件覆盖所有规定任务和用户目标的程度 |
| 功能正确性 | 软件产出正确结果的精度和准确度 |
| 功能适切性 | 功能促进完成特定任务的程度 |

**与开源运营的关联**：功能适合性对应你的"需求/缺陷关闭率"——关闭率高但关闭质量差（功能不完整或错误修复）同样是问题。

#### 特性二：性能效率（Performance Efficiency）

| 子特性 | 定义 | 典型度量 |
|--------|------|---------|
| 时间特性 | 功能执行的响应时间、处理速度 | P99 延迟、吞吐量 |
| 资源利用性 | 执行功能时使用资源的量 | CPU/内存占用率 |
| 容量 | 参数上限（用户数、数据量等）| 并发用户数上限 |

#### 特性三：兼容性（Compatibility）

| 子特性 | 定义 |
|--------|------|
| 共存性 | 与其他软件共享环境时的干扰程度 |
| 互操作性 | 与其他系统交换和使用信息的能力 |

#### 特性四：易用性（Usability）

包含 6 个子特性：可辨识性、易学性、易操作性、用户差错防御性、用户界面舒适性、易访问性。

**与开源运营的关联**：开源项目的"易用性"不仅是最终用户的体验，还包括**贡献者的开发体验（DX）**。一个 CONTRIBUTING.md 写得清晰、本地环境搭建简单的项目，新贡献者的转化率显著更高。

#### 特性五：可靠性（Reliability）

| 子特性 | 定义 | 典型度量 |
|--------|------|---------|
| 成熟性 | 正常运行时满足可靠性需求的程度 | MTTF（平均无故障时间）|
| 可用性 | 需要时可运行和可访问的程度 | SLA / Uptime |
| 容错性 | 故障时维持规定性能水平的能力 | MTTR（平均修复时间）|
| 易恢复性 | 故障后恢复数据和功能的能力 | RTO / RPO |

**与你的指标关联**：你的"软件有效维护率"和"版本稳定发布"指标是可靠性的运营侧观测，但缺少技术侧的 MTTF/MTTR 数据作为支撑。

#### 特性六：信息安全性（Security）

| 子特性 | 定义 |
|--------|------|
| 保密性 | 只有授权用户可访问数据 |
| 完整性 | 防止未经授权的访问或修改 |
| 不可抵赖性 | 行为的可证明性 |
| 可核查性 | 用户行为可被追溯 |
| 真实性 | 身份的可验证性 |

#### 特性七：可维护性（Maintainability）

这是与开源运营**最直接相关**的质量特性：

| 子特性 | 定义 | 典型度量 |
|--------|------|---------|
| 模块化 | 组件独立性，一处修改不影响其他 | 模块耦合度 |
| 可复用性 | 组件可被其他系统使用的程度 | API 覆盖率 |
| 易分析性 | 诊断缺陷或识别变更影响的难易程度 | 代码复杂度（Cyclomatic）|
| 易修改性 | 无引入缺陷的情况下进行修改的能力 | 技术债务比率 |
| 易测试性 | 建立测试准则并验证的难易程度 | 测试覆盖率 |

**对运营的启示**：可维护性指标是**预测性指标**而非结果性指标——代码可维护性高的项目，未来的"需求/缺陷关闭率"和"首次响应时长"会更好。建议将代码复杂度、测试覆盖率作为工程健康的领先指标纳入监控。

#### 特性八：可移植性（Portability）

| 子特性 | 定义 |
|--------|------|
| 适应性 | 适应不同硬件/软件/使用环境的能力 |
| 易安装性 | 在特定环境中安装/卸载的成功率 |
| 易替换性 | 替换同等软件的能力 |

### 7.3 使用质量模型（补充）

2023 年更新版新增了**使用质量模型**，评估软件在特定使用情境下的质量，包含 5 个特性：有效性、效率、满意度、抗风险性、情境覆盖性。

**与你的 NSS 净满意度的关系**：NSS 对应"满意度"特性，但 ISO 标准的满意度还包含更细化的子维度（实用性、信任度、愉悦感、舒适感），可作为后续深化研究的参考。

### 7.4 局限性

- 作为通用标准，缺乏开源社区特有的维度（如贡献者体验、治理透明度）
- 实施评估需要专业认证机构，成本较高
- 部分子特性（如"愉悦感"）定义模糊，难以量化
- 更新周期慢（2011→2023），对新兴技术场景（AI 软件、容器化应用）覆盖不足

## 8. DORA Metrics — DevOps 工程效能标准

### 8.1 背景与研究基础

DORA（DevOps Research and Assessment）指标由 Google 旗下的 DevOps Research 团队提出，基于对全球 **32,000+ 开发者和组织的 8 年追踪研究**（State of DevOps Report，2014-2022），是目前**数据基础最扎实的工程效能度量体系**。

DORA 的核心发现是：软件交付效能与组织绩效（盈利能力、市场份额、客户满意度）存在**统计显著的正相关关系**，且高效能团队与低效能团队之间存在**数量级差异**，而非简单的程度差异。

### 8.2 四大核心指标详解

#### 指标一：部署频率（Deployment Frequency）

**定义**：生产环境的代码部署/发版频率

**行业分级（2023 数据）：**

| 等级 | 频率 | 占比 |
|------|------|------|
| 精英（Elite）| 按需发布（每天多次）| ~18% |
| 高（High）| 每天到每周一次 | ~27% |
| 中（Medium）| 每周到每月一次 | ~38% |
| 低（Low）| 每月到每半年一次 | ~17% |

**与你的"版本稳定发布"指标的关系**：你的指标衡量的是发版的**稳定性**（准时/延迟），DORA 衡量的是发版的**频率**。两者互补——频率代表交付能力，稳定性代表承诺可靠性。

**开源场景特殊性**：开源项目不一定有"生产环境"，可将此指标映射为**正式版本（Release）发布频率**。精英开源项目（如 Kubernetes）每年有多个小版本发布，加上持续的补丁版本。

#### 指标二：变更前置时间（Lead Time for Changes）

**定义**：从代码提交到该代码在生产环境运行的时间

**行业分级（2023 数据）：**

| 等级 | 时长 |
|------|------|
| 精英 | < 1 小时 |
| 高 | 1 天到 1 周 |
| 中 | 1 周到 1 个月 |
| 低 | > 1 个月 |

**与开源运营的关联**：变更前置时间反映了**代码审查流程的效率**。PR 从提交到合入的时间，是开源场景下最接近的映射指标。对贡献者体验影响极大——等待 2 个月才能合入 PR 的项目，鲜有贡献者愿意持续贡献。

**补充洞察**：精英团队之所以前置时间短，不是因为跳过审查，而是因为他们有更好的**小批量提交习惯**（每次 PR 改动范围小）和**自动化测试覆盖**（减少人工验证时间）。

#### 指标三：变更失败率（Change Failure Rate）

**定义**：导致生产故障（需要修复、回滚或紧急补丁）的部署比例

**行业分级（2023 数据）：**

| 等级 | 失败率 |
|------|--------|
| 精英 | 0-5% |
| 高 | 5-10% |
| 中 / 低 | 10-15% 或更高 |

**与开源场景的关联**：对于开源软件，变更失败率可映射为**发版后 Critical/P0 Bug 的比例**——即发版后需要紧急发补丁版本的频率。这是衡量工程质量过程控制的关键指标，但多数开源项目并未系统追踪此数据。

#### 指标四：服务恢复时长（Time to Restore Service）

**定义**：生产故障从发生到恢复的平均时长（MTTR）

**行业分级（2023 数据）：**

| 等级 | 时长 |
|------|------|
| 精英 | < 1 小时 |
| 高 | < 1 天 |
| 中 | 1 天到 1 周 |
| 低 | > 1 周 |

**与开源场景的关联**：映射为**Critical Bug 从报告到修复发版的时长**，直接体现维护团队对严重问题的响应能力。

### 8.3 DORA 的重大发现：速度与稳定性不是矛盾的

DORA 研究中最反直觉、最有价值的发现是：**高频率发布的团队同时拥有更低的变更失败率**。

传统观念认为"发得越快，出问题越多"，但数据显示精英团队是唯一一个在所有四个指标上都优于其他团队的群体。原因在于：频繁小批量发布本身就是一种风险控制机制——每次变更范围小，问题容易定位和回滚。

对开源运营的启示：**鼓励小粒度、高频率的 PR，而非积压大 PR**，不仅提升贡献者体验，也降低工程风险。

### 8.4 2023 年新增指标：可靠性（Reliability）

DORA 2023 年版本在原有四指标基础上新增了**可靠性**作为第五维度，包括：
- SLO（服务质量目标）的定义和遵从率
- 错误预算的使用率
- 变更管理流程的成熟度

这一新增反映了行业从"交付效能"向"全链路可靠性"的关注迁移。

### 8.5 局限性

- 原设计面向内部研发团队，开源社区的映射需要一定转译
- "部署频率"在嵌入式、系统软件等领域的映射更为复杂
- 四个指标均为**滞后指标**（Lagging Indicators），只能反映结果，不能预测未来
- 行业 Benchmark 数据以西方科技公司为主，中国企业的对标适用性需谨慎

## 9. OpenSSF Scorecard — 开源供应链安全评估

### 9.1 背景与重要性

2020 年 SolarWinds 供应链攻击事件震惊全球，2021 年 Log4Shell 漏洞影响数百万系统，这两起事件将**开源软件供应链安全**推到了前所未有的高度。

OpenSSF（Open Source Security Foundation）由 Google、Microsoft、Linux 基金会等于 2020 年联合成立，Scorecard 是其核心项目之一，旨在提供**自动化的开源安全实践评分**。

Scorecard 的核心价值在于：**完全自动化，零人工成本，可集成到 CI/CD 流水线**，适合大规模批量评估。

### 9.2 20 项安全检查详解

Scorecard 当前版本（v4.x）包含约 20 项自动化检查，分为以下几类：

#### 源代码保护类

| 检查项 | 权重 | 评估内容 |
|--------|------|---------|
| Branch-Protection | 高 | 主干分支是否有保护规则（禁止强制推送、要求 PR 审查）|
| Code-Review | 高 | 合入主干的代码是否经过人工审查 |
| Maintained | 高 | 项目在过去 90 天内是否有活跃维护（commits 或 issue 活动）|
| Contributors | 中 | 是否有来自多个组织的贡献者 |

#### 依赖管理类

| 检查项 | 权重 | 评估内容 |
|--------|------|---------|
| Dependency-Update-Tool | 高 | 是否使用 Dependabot、Renovate 等自动依赖更新工具 |
| Pinned-Dependencies | 高 | CI 中的依赖是否固定到特定 hash（防止供应链污染）|
| Token-Permissions | 高 | GitHub Actions 的 Token 权限是否遵循最小权限原则 |

**深度解读**：依赖固定（Pinned Dependencies）是开源供应链安全中最被忽视的实践。很多项目在 CI 中使用 `uses: actions/checkout@main`，这意味着如果 actions/checkout 被攻击者控制，恶意代码会自动注入你的构建过程。正确做法是 `uses: actions/checkout@abc123def456`（固定到 commit hash）。

#### 构建安全类

| 检查项 | 权重 | 评估内容 |
|--------|------|---------|
| Binary-Artifacts | 高 | 代码库中是否存在二进制文件（潜在的恶意代码藏身之处）|
| SAST | 中 | 是否配置了静态应用安全测试工具（CodeQL、Semgrep 等）|
| Fuzzing | 中 | 是否配置了模糊测试（OSS-Fuzz 等）|
| Packaging | 中 | 发布包是否通过安全的自动化流程构建（而非手动）|

#### 漏洞管理类

| 检查项 | 权重 | 评估内容 |
|--------|------|---------|
| Vulnerabilities | 高 | 是否有未修复的已知 CVE 漏洞 |
| Security-Policy | 中 | 是否有 SECURITY.md 文件定义安全漏洞报告流程 |
| Signed-Releases | 中 | 发版产物是否有加密签名 |

#### 最佳实践类

| 检查项 | 权重 | 评估内容 |
|--------|------|---------|
| CII-Best-Practices | 中 | 是否通过 OpenSSF Best Practices Badge 认证 |
| License | 低 | 是否有明确的 License 文件 |
| Dangerous-Workflow | 高 | CI/CD 配置中是否有高风险的工作流模式 |

### 9.3 Scorecard 评分解读

Scorecard 对每个检查项给出 0-10 分，加权汇总为综合分。**官方建议关注每项的详细原因，而非仅看总分**。

典型评分示例：
- Linux Kernel：8.0/10（大型项目，安全成熟度高）
- 新孵化项目：3-4/10（安全基础设施尚未建立，属正常）
- 废弃项目：1-2/10（无维护，多项检查失败）

### 9.4 SLSA 框架：供应链完整性的进阶标准

与 Scorecard 配合使用的还有 **SLSA（Supply Chain Levels for Software Artifacts，供应链安全等级框架）**，由 Google 提出，定义了四个等级：

| 等级 | 要求 | 意义 |
|------|------|------|
| SLSA 1 | 有文档化的构建流程 | 基础可审计性 |
| SLSA 2 | 使用受控构建服务（如 GitHub Actions）| 防止手工篡改 |
| SLSA 3 | 构建服务有强安全保障，源代码可验证 | 防止构建服务被攻破 |
| SLSA 4 | 双人审查，密封构建（Hermetic Build）| 最高安全保障 |

目前大多数成熟开源项目处于 SLSA 2-3 级，SLSA 4 的实施成本极高，仅少数关键基础设施项目达到。

### 9.5 与你的"软件供应链安全（证书个数）"指标的关系

你当前使用"证书个数"作为供应链安全的度量，这是一个结果性指标。建议补充以下**过程性指标**作为领先指标：
- Scorecard 综合分（自动化，反映安全实践成熟度）
- 关键依赖的漏洞数量（CVE 数量及严重程度分布）
- SECURITY.md 完备性（是否有安全漏洞响应流程）

### 9.6 局限性

- 目前主要支持 GitHub 托管项目，对 Gitee、GitLab 的支持有限
- 自动化检查存在误报（如某些合理的 Binary-Artifacts 被标记为风险）
- 分数高不等于安全——Scorecard 只评估**安全实践**，不评估代码逻辑漏洞
- 对非 CI/CD 密集型项目（如文档项目、数据集项目）的适用性较弱

## 11. 与现有指标体系的 Gap 分析

### 11.1 现有指标体系优势评估

基于上述框架分析，你们现有的 13 个指标体系具有以下显著优势：

**覆盖面广**：从工程质量（维护率、发版稳定性）到社区治理（大象系数、多样性）再到影响力（搜索指数、下载量）和用户满意度（NSS），在单一指标集中实现了相对完整的覆盖，与 CHAOSS 的五大工作组有较高的对应度。

**创新指标**："贡献者撬动比（华为:其他）"是业界少见的原创指标，直接度量创始组织与外部社区的力量对比，比大象系数更具方向感（显示权力转移的进展），值得推广。

**综合健康度**：将多指标聚合为单一健康度分的设计，便于管理层快速把握全局，与 OSS-Compass 的设计思路一致。

### 11.2 关键缺口识别

对照业界框架分析，现有体系存在以下主要缺口：

#### 缺口一：缺乏领先指标（代码可维护性）

**问题**：现有指标多为**滞后指标**（反映已发生的结果），缺乏能预测未来健康状况的**领先指标**。

**建议补充：**
- 测试覆盖率（Test Coverage）：预测未来的缺陷关闭率
- 代码复杂度（Cyclomatic Complexity 均值）：预测未来的维护成本
- 技术债务比率（SonarQube 等工具可自动生成）

#### 缺口二：缺乏生命力预警指标

**问题**：现有体系能反映当前状态，但缺少**衰退预警**信号——即一个健康的社区开始走向衰退时，早期信号是什么。

**建议补充：**
- 活跃贡献者留存率（老贡献者流失速度）
- 新贡献者转化率（首次 PR → 第二次贡献的比例）
- 维护者负载（平均每个维护者处理的 Issue/PR 数量，过高是预警信号）

#### 缺口三：缺乏工程安全类指标精度

**问题**："软件供应链安全（证书个数）"是结果性指标，缺少过程性的安全实践度量。

**建议补充：**
- OpenSSF Scorecard 分（可自动化获取）
- 高危 CVE 漏洞修复时长（从 CVE 发布到修复发版的时间）
- 关键依赖的未修复漏洞数量

#### 缺口四：缺乏贡献质量维度

**问题**：现有指标关注贡献**数量**（交互率、多样性），但未度量贡献**质量**。

**建议补充：**
- PR 通过率（提交 PR 中最终被合入的比例，反映贡献门槛是否合理）
- 代码审查参与率（有 Review 评论的 PR 占比）
- Issue 复现率（Issue 报告中信息完整可复现的比例）

#### 缺口五：缺乏生命周期定位标签

**问题**：现有指标是平铺的列表，没有生命周期关联，导致不同阶段社区的横向对比失真。

**建议**：为每个社区建立显式的**生命周期标签**（孵化/成长/成熟/持续），并在报告中以阶段内对比替代跨阶段对比。具体方法见第 12 节。

### 11.3 指标优化建议总表

| 现有指标 | 评价 | 优化建议 |
|---------|------|---------|
| 软件有效维护(%) | 定义需明确化 | 补充"有效维护"的判定标准（如：commit 频率 + Issue 响应率的组合条件）|
| 版本稳定发布(天) | 结果性指标，好 | 补充"发版后 P0 Bug 率"作为质量维度 |
| 需求/缺陷关闭率(%) | 需区分需求与缺陷 | 拆分为需求关闭率和缺陷关闭率，两者权重不同 |
| 贡献者交互率 | 好，较有创意 | 补充"首次贡献者的 PR 被回复率"细化新人体验 |
| 首次响应时长(天) | 核心指标，保留 | 建议分拆：Issue 首次响应 vs PR 首次响应，两者意义不同 |
| 贡献组织多样性(个) | 好 | 补充 HHI 指数（赫芬达尔-赫希曼指数），比个数更能反映分布均衡性 |
| 大象系数(个) | 好，与业界对齐 | 结合巴士系数（个人层面）形成完整的集中度评估 |
| 贡献者撬动比 | 创新指标，价值高 | 建议明确目标值（如 2 年内从 2.51 降至 1.5 以下）|
| 软件供应链安全(证书个数) | 结果性不足 | 补充 OpenSSF Scorecard 分作为过程性指标 |
| 技术影响力(搜索指数) | 较粗糙 | 结合 Stack Overflow 提及量、GitHub 依赖数等多维度 |
| NSS 净满意度(%) | 好，但采集难度高 | 明确调研频率和样本代表性方法论 |
| 下载量(万次) | 好，但需去噪 | 区分真实用户下载 vs CI 自动下载（后者会大幅虚高数据）|
| 健康度(平均值) | 简单平均存在掩盖效应 | 考虑加权平均，或改用雷达图呈现结构而非单值 |

## 12. 基于生命周期的综合度量建议

### 12.1 生命周期判定方法

建议采用以下**量化门槛**判定社区所处阶段，避免主观判断：

| 维度 | 孵化期 | 成长期 | 成熟期 | 持续期 |
|------|--------|--------|--------|--------|
| 贡献组织数 | ≤ 3 | 4-9 | ≥ 10 | ≥ 10（但增速放缓）|
| 大象系数 | N/A | 1-2 | ≥ 3 | ≥ 3（但需关注老化）|
| 下载量（年）| < 10 万次 | 10-500 万次 | > 500 万次 | 稳定或下降 |
| 活跃贡献者 | < 10 | 10-50 | > 50 | > 30 但趋势下降 |
| 版本历史 | pre-1.0 | 1.x | 2.0+ 或 3 年以上 | 5 年以上，API 稳定 |

**判定规则**：5 个维度中，满足 3 个即可判定为该阶段。存在争议时，以**贡献组织数**和**下载量**为主要依据。

### 12.2 各阶段核心度量卡

#### 孵化期度量卡

**核心目标**：验证技术方向，建立工程规范，留住第一批外部贡献者

| 优先级 | 指标 | 目标值（参考）|
|--------|------|-------------|
| P0 | 首次响应时长 | < 3 天 |
| P0 | 需求/缺陷关闭率 | > 60%（建立基本信任）|
| P0 | 版本稳定发布 | 季度发版节奏稳定 |
| P1 | 贡献者交互率 | PR 均有回复 |
| P1 | OpenSSF Scorecard | > 4 分（建立基础安全规范）|
| 观测 | 下载量增速 | 月环比 > 10% 即为健康 |
| 不适用 | 大象系数、撬动比 | 此阶段不应施压 |

#### 成长期度量卡

**核心目标**：扩大用户规模，引入外部力量，证明商业价值

| 优先级 | 指标 | 目标值（参考）|
|--------|------|-------------|
| P0 | 下载量增速 | 年同比 > 30% |
| P0 | 技术影响力（搜索指数）| 持续上升趋势 |
| P0 | 贡献组织多样性 | 每季度新增 1-2 个组织 |
| P0 | NSS 净满意度 | > 50%（NPS 参考：口碑传播临界点）|
| P1 | 贡献者撬动比 | 创始组织占比每年下降 5%+ |
| P1 | 首次响应时长 | < 2 天（规模扩大后更需保持）|
| 观测 | 大象系数 | ≥ 2 即合格，不苛求 |

#### 成熟期度量卡

**核心目标**：实现社区自治，建立生态系统，防止单一组织控制

| 优先级 | 指标 | 目标值（参考）|
|--------|------|-------------|
| P0 | 大象系数 | ≥ 3（CNCF 毕业标准）|
| P0 | 贡献者撬动比 | < 1.5（创始组织不超过总贡献的 60%）|
| P0 | 贡献组织多样性 | HHI 指数 < 0.25（分散度健康）|
| P0 | 软件供应链安全 | OpenSSF Scorecard > 7 分 |
| P1 | 健康度综合分 | > 70 分（OSS-Compass 基准）|
| P1 | NSS 净满意度 | > 60%（成熟项目的用户黏性）|
| 观测 | 下载量 | 关注稳定性，不再追求高增速 |

#### 持续期度量卡

**核心目标**：防止衰退，保持技术相关性，培育新一代维护者

| 优先级 | 指标 | 目标值（参考）|
|--------|------|-------------|
| P0 | 软件有效维护率 | > 90%（核心防线）|
| P0 | 活跃贡献者留存率（新增）| > 70%（老贡献者留存）|
| P0 | 首次响应时长 | < 5 天（老社区响应变慢的预警线）|
| P1 | 新贡献者转化率（新增）| > 30%（防止社区老化封闭）|
| P1 | 技术影响力 | 保持稳定，关注同类新兴项目的崛起 |
| P1 | 需求/缺陷关闭率 | > 70% |
| 观测 | 健康度综合分 | 设置下限预警（< 60 分触发专项治理）|

### 12.3 跨阶段对比的标准化方法

为了在同一报告中同时展示不同阶段项目，建议采用**阶段内 Z-Score 标准化**方法：

**步骤：**
1. 将所有社区按生命周期阶段分组
2. 对每个阶段内的指标计算均值（μ）和标准差（σ）
3. 每个社区的指标得分 = (原始值 - μ) / σ
4. Z-Score > 1：该阶段内表现优秀；Z-Score < -1：需要关注

这样，孵化期社区的"首次响应时长 1 天"和成熟期社区的"首次响应时长 1 天"，虽然绝对值相同，但在各自阶段内的**相对意义**是不同的。

### 12.4 阶段跃迁评估机制

建议每半年对所有社区进行**阶段跃迁评估**，判断是否应升级到下一阶段或触发预警。

**跃迁条件（满足其中 3/5）：**

孵化期 → 成长期：
- 贡献组织数 ≥ 4
- 下载量突破 10 万次
- 有至少 1 个外部组织的生产使用案例
- 首次响应时长稳定 < 3 天
- 获得社区自发的外部技术文章/演讲引用

成长期 → 成熟期：
- 贡献组织数 ≥ 10
- 大象系数 ≥ 3
- 贡献者撬动比 < 2（外部贡献超过创始组织贡献）
- 有公开的 GOVERNANCE 治理文档
- NSS 净满意度 > 60%
