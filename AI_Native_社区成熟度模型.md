# AI Native 开源社区成熟度模型

> 衡量一个开源社区对"使用 AI 工具的开发者"的服务能力

---

## 模型设计思路

业界目前没有专门针对"开源社区 AI Native 程度"的标准化评估体系。
参考 CMMI、云原生成熟度模型（CNCF）、DevEx 框架等设计思路，
结合开源社区的实际运作特点，提出以下五维度评估模型。

---

## 五大维度定义

### 维度一：项目可读性（Project AI-Readability）
**衡量**：社区的开源项目是否结构化到让 AI 工具能准确理解和描述

| 分值 | 标准描述 |
|------|----------|
| 1 | 无 README 或 README 内容极少；无贡献指南 |
| 2 | 有基础 README，但结构混乱；CONTRIBUTING.md 缺失或过于简单 |
| 3 | README 结构完整（功能/安装/使用）；有 CONTRIBUTING.md |
| 4 | 有 ARCHITECTURE.md 或等效设计文档；有 CHANGELOG；CONTRIBUTING 包含具体操作步骤 |
| 5 | 有 CLAUDE.md / copilot-instructions / .cursorrules 等 AI 上下文文件；文档定期更新有专人负责 |

**评估方式**：抽查社区 Top 10 活跃项目，按上述标准打分取平均值

---

### 维度二：贡献流程 AI 化（AI-Augmented Contribution）
**衡量**：AI 工具能在多大程度上辅助开发者完成完整的贡献流程

| 分值 | 标准描述 |
|------|----------|
| 1 | 无任何 Issue 模板；PR 无模板；无自动化检查 |
| 2 | 有基础 Issue/PR 模板；有简单 CI 检查 |
| 3 | Good First Issue 有规范标签；PR 模板包含 checklist；CI 覆盖测试和 Lint |
| 4 | Good First Issue 描述详细（背景/文件路径/完成标准）；有 PR 示例集；AI Review Bot 试点 |
| 5 | Good First Issue 附带"AI 辅助提示"；AI Review Bot 全覆盖；PR 平均 Review 时间 < 48h |

**评估方式**：随机抽查 20 个 Good First Issue 评分；检查 CI/CD 配置；统计 PR Review 时间

---

### 维度三：开发者体验（Developer Experience for AI Users）
**衡量**：使用 AI 编程工具的开发者在社区中的上手体验

| 分值 | 标准描述 |
|------|----------|
| 1 | 无专门的新手引导；新人只能自行摸索 |
| 2 | 有基础新手文档，但未针对 AI 工具使用者设计 |
| 3 | 新手引导中有 AI 工具相关内容（推荐工具/配置方法）；有 Buddy/Mentor 机制 |
| 4 | 有针对"AI 辅助贡献"的专属上手路径；提供 Prompt 参考或工具配置模板 |
| 5 | 全流程无缝：新人用 AI 工具可在 2 小时内完成首次贡献；有实证数据支撑 |

**评估方式**：实际走一遍新手流程（"黑盒测试"）；新人满意度调查；首次贡献转化率数据

---

### 维度四：内容可发现性（Content Discoverability by AI）
**衡量**：社区的文档和内容被 AI 工具检索、引用、推荐的能力

| 分值 | 标准描述 |
|------|----------|
| 1 | 文档散乱，无结构；无独立文档站点 |
| 2 | 有文档站点，但内容更新滞后；无 SEO 优化 |
| 3 | 文档站点内容完整、结构清晰；有基础 SEO；GitHub Topics 标签规范 |
| 4 | 文档内容密度高（少空话、多具体步骤和数据）；适配 AI 搜索引擎的结构化写作 |
| 5 | 支持 llms.txt 协议；有证据显示内容被 Perplexity/ChatGPT Search 等引用；定期验证 AI 推荐准确性 |

**评估方式**：在主流 AI 搜索工具中搜索社区相关问题，验证是否被正确引用；检查文档结构质量

---

### 维度五：社区 AI 文化（Community AI Culture）
**衡量**：社区对 AI 辅助开发的包容度、推广力度和贡献多样性

| 分值 | 标准描述 |
|------|----------|
| 1 | 无明确立场；部分 Maintainer 排斥 AI 生成贡献 |
| 2 | 默许 AI 辅助贡献，但无明确政策；没有相关活动 |
| 3 | 有明确的 AI 辅助贡献政策（允许并注明质量要求）；每季度有 AI 相关活动 |
| 4 | 积极推广 AI 辅助贡献；贡献认可体系包含 AI 时代新型贡献形式（文档/翻译/测试）|
| 5 | AI 时代新开发者（1-3年经验）占月新增贡献者 ≥ 30%；有可量化的多样性提升数据 |

**评估方式**：社区政策文档检查；贡献者背景调查；活动频次统计；贡献者构成数据

---

## 成熟度等级定义

总分 = 五个维度分值相加（满分 25 分）

| 等级 | 分值区间 | 描述 |
|------|----------|------|
| **L1 - 初始级** | 5-8 | 尚未意识到 AI Native 建设的必要性，基础薄弱 |
| **L2 - 探索级** | 9-12 | 开始在部分项目或流程中尝试 AI 友好改造 |
| **L3 - 发展级** | 13-17 | 有系统性建设，大部分维度达到基本水平 |
| **L4 - 成熟级** | 18-21 | AI Native 理念全面落地，开发者体验显著提升 |
| **L5 - 引领级** | 22-25 | 行业标杆，能输出最佳实践影响更大生态 |

## 评估周期建议

| 评估类型 | 频率 | 执行者 | 用途 |
|----------|------|--------|------|
| 自评快照 | 每季度 | 社区运营团队 | 追踪进展，调整优先级 |
| 深度评估 | 每半年 | 内部 + 外部视角结合 | 发现盲点，对标行业 |
| 公开报告 | 每年 | 社区委员会 | 建立透明度，吸引贡献者 |

## 与现有框架的关系

| 现有框架 | 关系 |
|----------|------|
| CHAOSS 社区健康度指标 | 互补：CHAOSS 关注社区健康（活跃度/多样性），本模型专注 AI 友好度 |
| CNCF 云原生成熟度 | 方法论借鉴：五维度 + 分级描述的结构设计 |
| GitHub DevEx 报告 | 数据参考：开发者体验的行业基准数据 |
| ThoughtWorks Tech Radar | 策展理念借鉴：权威评估 + 定期更新 |
